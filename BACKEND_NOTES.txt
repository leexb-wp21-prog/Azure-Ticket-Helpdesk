Backend Integration Notes (Create Ticket / My Tickets)
=====================================================

Purpose
-------
This file centralizes all frontend-to-backend integration expectations so backend can quickly align implementation.


1) Required API Endpoints
-------------------------
1. POST /api/submit_ticket
   - Used by create ticket form submit.
   - Frontend sends JSON payload.

2. GET /api/get_tickets?email=<requester_email>
   - Used by "My Tickets" list retrieval.
   - Should return requester's tickets.


2) Current Submit Payload (Frontend)
-----------------------------------
The frontend currently submits:
- request_type: string   (mapped from "Ticket type" field)
- name: string
- email: string
- category: string       (currently mapped from "Assign to" field value)
- department: string     (same source as category for now)
- priority: "High" | "Medium" | "Low" (or future "Urgent")
- assigned_to: string
- subject: string        ("Ticket Name")
- location: string       (optional, currently hidden in UI)
- description: string

Note:
- Field mapping is based on current UI decisions and can be normalized later if backend wants stricter schema.


3) Expected Response Shape
--------------------------
POST /api/submit_ticket should return a ticket object containing at least:
- ticket_id
- status
- submitted_at (or created_at)
- updated_at
- subject
- priority
- category
- department (recommended)
- assignedTo or assigned_to
- description

GET /api/get_tickets should return:
{
  "tickets": [ ...ticket objects... ]
}


4) Consistency Requirement
--------------------------
After POST succeeds, GET by same requester email should include the newly created ticket quickly.
Frontend "My Tickets" treats GET as source of truth for persisted data.


5) Temporary Frontend Preview (TO REMOVE)
-----------------------------------------
Frontend currently contains a temporary local preview bridge so user can see new ticket immediately after create.

Markers in code:
- REMOVE_BEFORE_BACKEND_READY

Current temporary behavior:
- Inject newly created ticket into local list immediately
- Auto-switch to "My Tickets" panel for instant visual feedback

Backend-ready cleanup:
- Remove the local preview injection path marked with REMOVE_BEFORE_BACKEND_READY
- Keep only backend-driven refresh flow


6) Optional Improvements for Backend + Frontend Alignment
--------------------------------------------------------
- Return a canonical field contract (e.g., category vs department vs assigned_to) to avoid duplicate mapping.
- Provide stable enum values for status and priority.
- Include comments/timeline in ticket detail response if available.

- Manage Tickets, Analytics, and Support Teams are not range-gated in frontend routing.


8) Admin Overview Interactive Table (Current Temporary Behavior)
----------------------------------------------------------------
Overview table now intentionally supports interactive controls with temporary frontend-only logic:
- `Status` is a dropdown (Open / In Progress / Resolved)
- `Assigned Team` is a dropdown
- `View` button is clickable (temporary alert preview)

Temporary behavior in code:
- Updates local in-memory row state only
- Recalculates overview metrics instantly after status changes
- Does not persist to backend yet

Code markers for backend handover:
- `TODO_BACKEND_REAL_DATA` in `frontend/admin.js`

Recommended backend endpoints for this table:
1. PATCH /api/admin/tickets/{ticketId}/status
   Request: { "status": "Open|In Progress|Resolved" }
   Response: updated ticket object

2. PATCH /api/admin/tickets/{ticketId}/assignment
   Request: { "assignedTeam": "<team_name>" }
   Response: updated ticket object

3. GET /api/admin/tickets/{ticketId}
   Used by `View` action for real ticket detail screen/modal.

Admin overview latest behavior update:
- `View` now opens a real ticket detail modal in frontend (no browser alert).
- `Status` and `Assigned Team` dropdown values are committed only when admin clicks `Update`.
- `Update` currently persists to temporary frontend state immediately, then attempts backend PATCH if API is available.
- For local verification, admin mock tickets now include spread timestamps:
  - 2 tickets created today
  - 2 additional tickets earlier this week
  - 2 additional tickets earlier this month
  - 1 additional ticket earlier this year
  - 1 ticket outside current year (to validate year filter exclusion)

Backend team action:
- Return consistent `ticketId` in admin endpoints so PATCH routes can map rows reliably.
- Add ticket detail payload enrichment for modal (requester contact, timestamps, notes, SLA fields) when ready.


9) Admin Team Management - Add Team UI + Error Handling
-------------------------------------------------------
Frontend now uses a proper Add Team modal (not browser prompt):
- Fields: `team name` (required), `team lead` (optional), `team lead email` (optional)
- Inline validation errors:
  - team name required
  - team name duplicate
  - email format validation (if provided)
- Clear user feedback via toast when validation fails or create succeeds/falls back

Temporary data behavior (for testing before backend is ready):
- New team is inserted into local frontend state first (`TEMP_DATA` marker in `frontend/admin.js`)
- Then frontend attempts backend POST sync
- If POST fails, team stays available locally for the current session

Backend endpoint for real create:
1. POST /api/admin/support_teams
   Request:
   {
     "id": string,
     "name": string,
     "badge": string,
     "badgeClass": string,
     "members": number,
     "activeTickets": number,
     "lead": string,
     "leadRole": string,
     "email": string,
     "phone": string,
     "permissions": number,
     "stats": { "active": number, "resolved": number, "avgTime": string, "satisfaction": string },
     "staffMembers": [ { "name": string, "role": string, "email": string, "phone": string, "activeTickets": number } ]
   }
   Response:
   - 201 with created team object (preferred)
   - 409 on duplicate name
   - 400 on validation failures (with field-level error payload when possible)
